LLVM has included a Fortran compiler, Flang, since LLVM 11. If that was not news
to you you might also know that it was not called `flang` but instead `flang-new`.

The LLVM 20 release brings an end to the era of `flang-new`. The
community has decided it has reached a level of quality such that it deserves a
new name.

The "new" name? That's right, `flang`.

This might seem insignificant but for the Flang community it is a big achievement 
after many years of work. From Flang's origins in the Portland Group, becoming open
source, acceptance into LLVM and being completely redesigned along the way.

So if you want to know how we got here, this article is for you.

If you to try out Flang 20, you can download it now or try it right
in your browser using Compiler Explorer.

# Why Fortran?

The majority of this article won't require any Fortran knowledge, but it's
important to know the context the language was created in, the context it has
existed in for the last 70 years and why people keep investing in it.

IBM has an excellent [article](https://www.ibm.com/history/fortran) on the history
of the language and I encourage you to look at the
["Programmer's Primer for Fortran"](https://archive.computerhistory.org/resources/text/Fortran/102653987.05.01.acc.pdf) as well.

The name Fortran was created from "Formula Translation" and reading those
manuals you will see why. A focus on the mathematical use case, freeing programmers
from writing machine specific assembly code.

Instead they could write code that looked like their formulas. Something we expect
today but at the time a revolution and one I certainly feel when watching those
original demonstrations.

This formula focus led to Fortran being used heavily in scientfic computing as
the years went on. Weather modelling, fluid dynamics, computational chemistry,
just to name a few.

A 2023 [study](https://arxiv.org/pdf/2308.13274) put it this way:
> Whilst many alternative programming languages have come
> and gone, it [Fortran] has regained its popularity for writing high
> performance codes. Indeed, over 80% of the applications
> running on ARCHER2, a 750,000 core Cray EX which is
> the UK national supercomputer, are written in Fortran.

**Note:** The term "codes" in a Fortran context usually means "programs".

# Did We Need Another Fortran Compiler? 

There are and have been many Fortran compilers. Some are vendor specific 
such as the
[Intel Fortran Compiler](https://www.intel.com/content/www/us/en/developer/tools/oneapi/fortran-compiler.html)
or NVIDIA's [HPC compilers](https://developer.nvidia.com/hpc-compilers). Then
there are open source options like [gfortran](https://gcc.gnu.org/fortran/)
(see [here](https://fortran-lang.org/en/compilers/) for a comprehensive list).

So why build another one? In 2020 Gary Klimowicz (NVIDIA)
[laid out](https://www.youtube.com/watch?v=Fy68k5hHgLk) the goals for Flang,
which I've paraphrased here:

* Develop Flang under a permissive license like LLVM's, which is more palletable
  to commercial users and contributors.
* Develop an active community of Fortran compiler developers that includes
  companies and institutions.
* Support Fortran tool development by being based on existing LLVM frameworks
  that tools already support.
* Support Fortran language experimentation for future language standards proposals.

The last one can be explained by comparison to Clang. For proposals of new features
for the C++ language, a common path is to implement them on top of either Clang
or GCC to prove they will work. However the two compilers have different strengths.
What is easy for GCC might not be for Clang and vice versa.

Therefore having both allows us to find those problems early. And so, we hope,
will having Flang and gfortran.

Finally, though Fortran is an ISO standard we know that standards can be ambiguous
and implementations can make mistakes. Building a new compiler is a chance to
rethink those decisions and find corners of the standards that have been neglected.

# PGI to LLVM, the Flang Story

As you will see, stretching the Flang story all the way back to 1989 is a bit
of a cheat. However, The Portland Group's history with supercomputing and US
National Labs is important context, so I have included it.

* **1989** The Portland Group aka
  [PGI](https://web.archive.org/web/19970628161656/http://www.pgroup.com/corp_home.html) 
  is formed. To provide C, Fortran 77 and C++ compilers for the new Intel i860
  market.
* **1990** Intel bundles PGI's compilers with its iPSC/860 supercomputer.
* **1996** PGI
  [works with](https://web.archive.org/web/20100528051556/http://www.sandia.gov/ASCI/Red/papers/Mattson/OVERVIEW.html)
  Sandia National Laboratories to provide compilers for the ASCI Option Red
  supercomputer.
* **December 2000** PGI becomes a
  [wholly owned subsiduary](https://www.electronicsweekly.com/news/archived/resources-archived/stmicroelectronics-to-acquire-pgi-2000-12/) of
  STMicroElectronics. 
* **July 2013** PGI is [sold to NVIDIA](https://www.theregister.com/2013/07/30/nvidia_buys_the_portland_group/).

At this point we see the first steps towards what would become known as
"Classic Flang", the first iteration of Flang.

* **November 2015** NVIDIA is part of the US Department of Energy
  Exascale Computing Project. Including a commitment to create an open source
  [Fortran compiler](https://www.llnl.gov/article/41756/nnsa-national-labs-team-nvidia-develop-open-source-fortran-compiler-technology).

  > “The U.S. Department of Energy’s National Nuclear Security Administration and its
  > three national labs have reached an agreement with NVIDIA's PGI division to adapt
  > and open-source PGI's Fortran frontend, and associated Fortran runtime library,
  > for contribution to the LLVM project.”
* **May 2017** The first source code release of Classic Flang as a separate
  respoitory outside of LLVM itself. This is when other companies began to be
  involved, such as Arm and AMD.
* **April 2018** Steve Scalpone (NVIDIA) [announces](https://www.youtube.com/watch?v=sFVRQDgKihY)
  at Euro LLVM 2018 that the frontend (which was PGI's frontend at the time)
  will be re-written to address feedback from the LLLVM community. This is what
  became known as "F18".

  > “We started writing this about 30 years ago, and our data structure is called integer.”
  - Steve Scalpone, on the subject of the PGI frontend.

  There was also a hint at a rewrite of the backend, more on this later.
* **April 2019** F18 is [approved for migration](https://discourse.llvm.org/t/f18-is-accepted-as-part-of-llvm-project/51719)
  into the LLVM Project monorepo.

  Note that although "F18" is what was approved, the frontend F18 moved into
  `flang` in the monorepo and would be followed later by a new backend.
* **August 2019** The [first appearance](https://github.com/flang-compiler/f18/commit/b6c30284e7876f6ccd4bb024bd5f349128e99b7c)
  of the `flang.sh` driver wrapper script, which will be important later.
* **December 2019** The [plan](https://discourse.llvm.org/t/flang-landing-in-the-monorepo/54022)
  for rewriting the F18 git history to fit into the LLVM project is announced.
  This effort was lead by Arm, with Peter Waller (Arm) going so far as to write
  a [custom tool](https://github.com/llvm/llvm-project/commit/55d5e6cbe2509a24132d056e1f361dc39312929b#diff-c389405236998090c7c8b9741506f01fb28abbd7da52e9566323c585ac0eb89cL910)
  to rewrite the history of F18.
* **Feburary 2020** The [plan](https://discourse.llvm.org/t/plan-for-landing-flang-in-monorepo/54546)
  for improvements to F18 to meet the standards required for inclusion in the
  LLVM monorepo is announced.
* **April 2020** Upstreaming of F18 into the LLVM Project monorepo as `flang` is
  [completed](https://github.com/llvm/llvm-project/commit/b98ad941a40c96c841bceb171725c925500fce6c).

At this point the frontend was in the LLVM monorepo, but it could not do any code
generation. Behind the scenes, work on the `fir-dev` branch had started to
address this.

From here on we get into the various names Flang has had. To make sense of these
choices, remember that Classic Flang was (and still is) in use. Its
binary was simply `flang`. Therefore it was decided that offering something
of the same name that was in fact not as capable, was not a good idea.

* **October 2020** Flang is included in an LLVM release for the first time in
  LLVM 11.0.0. At this time the driver part was called `f18` and the afformentioned
  script `flang.sh` was included.
* **October 2021** First appearance of the `flang-new` binary, in the LLVM 13.0.0
  release.
* **April 2022** NVIDIA [ceases development](https://discourse.llvm.org/t/nvidia-transition-from-fir-dev/61947)
  of the `fir-dev` branch, located back in the Classic Flang project. This branch
  adds code generation to F18. Upstreaming of `fir-dev` to the LLVM Project
  monorepo begins around this date.
* **April 2022** `flang-new` can now do code generation if an experimental flag
  is used. This is thanks to code being upstreamed from `fir-dev`.
* **May 2022** The `flang.sh` driver script becomes `flang-to-external-fc`. It
  now allows the user to use `flang-new` to process Fortran source code then send
  the result to an existing Fortran compiler such as gfortran.
  
  This may seem
  strange but it allowed users to drop in Flang as a replacement for an existing
  compiler and test Flang's parsing code on their own codebases without having to
  wait for code generation to be finished.
* **June 2022** `flang-new` changed to do code generation
  [by default](https://github.com/llvm/llvm-project/commit/0867d4157328169c570f4d1e9c01806624035873)
  with no extra flags required.
* **July 2022** NVIDIA [finish](https://discourse.llvm.org/t/nvidia-transition-from-fir-dev/61947/5)
  upstreaming `fir-dev`.

At this point we had what we know today as Flang. The F18 frontend and code
generation from the `fir-dev` branch. This combination was known as `flang-new`.

* **April 2024** The `flang-to-external-fc` script is removed. Marking the point
  where Flang was mature enough to be a drop in replacement for anyone willing
  to test their code with it.
* **October 2024** The community deems that Flang has met the criteria to not be
  "new" and the name is changed. Goodbye `flang-new`, hello `flang`!
